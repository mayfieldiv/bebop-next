## Architecture Notes

### Differences from Production Bebop Rust
bebop-next diverges significantly from the production `~/vendor/bebop` Rust generator:

| Aspect | Production | bebop-next |
|---|---|---|
| Generator language | C# (`RustGenerator.cs`) | Rust (self-hosted binary) |
| Plugin protocol | JSON via Chord extensions | Bebop-encoded binary on stdin/stdout |
| Owned types | Separate `mod owned { }` module | Inline `Cow` + `TypeOwned` alias |
| Timestamp | 64-bit .NET ticks (`Date`) | 12-byte (i64 seconds + i32 nanos) |
| UUID | Microsoft byte ordering (`Guid`) | Canonical byte-for-byte |
| Half-precision | Not supported | `f16` / `bf16` via `half` crate |
| Int128 | Not supported | `i128` / `u128` native |
| Duration | Not supported | 12-byte (i64 + i32) |
| Fixed arrays | Not supported | `[T; N]` with const generics |
| Zero-copy mechanism | `SliceWrapper` enum | `Cow<'buf, T>` |
| String wire format | u32 len + UTF-8 (no NUL) | u32 len + UTF-8 + NUL terminator |

### Key Design Decisions
- **`Cow` over dual modules**: Instead of generating two parallel type hierarchies (borrowed + owned), bebop-next uses `Cow<'buf, str>` inline. This halves the generated code and makes the owned/borrowed transition explicit via `into_owned()`.
- **Always-lifetime unions**: Unions always carry `'buf` due to the `Unknown(u8, Cow<'buf, [u8]>)` forward-compatibility variant.
- **NUL-terminated strings**: bebop-next adds a trailing NUL byte to strings on the wire, matching the whitepaper spec. Production bebop does not.
